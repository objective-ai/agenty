3 Approaches to Mission Mode

Same end goal, different trade-offs on complexity vs. polish

A

Static Mission Config + Tool Calls ⭐ Recommended

A mission-config.ts file defines the Dragon Bridge mission (topic, image asset key, stat labels). On page load, Cooper calls initMissionBoard tool to populate the big question. As the kid answers, Cooper calls updateStat to animate individual counters.

Pros
✓ Deterministic image loading
✓ Simple React state (no DB)
✓ Fast to build
✓ Easy to add missions later
Cons
✗ New mission = new config entry
✗ Stat labels must be pre-defined
✗ Not fully dynamic
Tool shape: updateStat({ label: "Area Calculated", value: 24975, unit: "m²", animate: true })
B

Fully Agent-Driven Board

No config file. Cooper calls initMissionBoard with everything on session start: mission title, big question, stat definitions, even which image asset key to use. Board is entirely Cooper's output.

Pros
✓ Zero config files
✓ Works with any uploaded PDF
✓ Stat labels are topic-aware
Cons
✗ Image asset key can hallucinate
✗ First message is slow (tool call + render)
✗ Harder to test
✗ More prompt engineering
C

Hybrid: Thin Config + Dynamic Stats

Config file owns: topic slug, image asset, stat label schema. Cooper owns: big question generation, stat values. On session start, the UI loads the config; Cooper's first message calls setBigQuestion. Subsequent answers call updateStat.

Pros
✓ Deterministic images
✓ Stat labels consistent
✓ Cooper feels alive
✓ Easiest to extend
Cons
✗ Split ownership (config vs AI)
✗ Slightly more moving parts
✗ Config + RAG must stay in sync
My recommendation: A — for v1 with the Dragon Bridge mission, a static config is faster to build and easier to get right. The tool call shape is the same as C/B, so you can migrate to hybrid later without changing the UI. Config-first, then go dynamic when you need it.
